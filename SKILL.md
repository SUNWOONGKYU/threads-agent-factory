# Threads Agent Factory

소셜(Threads)에서 사용자의 실수요를 발굴 → 당신의 카탈로그에 **없는** 유용한 에이전트를 **에이전트
제조 방법론**으로 제조 → **라이브 유용성 80점 통과분만 출시**하는 반자동 운영 스킬.
= 발굴·큐레이션 층 + 제조 코어 + 품질 게이트.

## ⚠️ 비용 원칙 (최우선·반드시 지킴)
- **LLM 추론은 구독 LLM으로 직접 한다 — API 크레딧 금지.** 대상: 아이디어 추출, 중복·gap 판정,
  유용성 80점 평가, system_prompt·소개 작성, 부품 적합성 판단, 후보 큐레이션.
- **API/외부 호출은 불가피한 것만**: ① 소셜 수집(스크레이퍼 액터) ② 부품 실다운로드·이미지/영상/음성
  생성·웹검색 ③ 프로덕션 런타임(출시 후 검증 1~2회).
- 즉 추론을 외부 LLM API 호출 스크립트로 돌리지 말 것. 그 판단은 이 스킬을 실행하는 운영자가 직접 한다.
  수집(스크레이퍼)만 스크립트로.

## 실행 환경
- 작업 디렉토리: 당신의 플랫폼/카탈로그 저장소. 환경변수에 소셜 스크레이퍼 토큰·DB 키·provider 키.
  **토큰·키 값은 저장소에 커밋하지 말 것 — 환경변수/시크릿 매니저로만.**
- 출시 대상이 변경 관리(예: Grid/이슈 트래커) 프로젝트라면, 출시 시 등재 의무를 따른다.

## 파이프라인 (7단계)

### ① 수집 (API: 소셜 스크레이퍼)
스크레이퍼 액터로 Threads 글 수집.
- 키워드 = **사용자 니즈·페인·실활용형** (예: "업무 자동화", "마케팅 자동화", "AI로 만들었", 도메인+자동화).
  ❌ 개발자 도구명(특정 CLI·프로토콜·개발 유행어)은 잡음 → 금지.
- `after` = 최신 N개월(기본 **3**), 인기순 정렬, 한국어 필터(한글 일정 길이 이상), 글 ID 중복제거.
- 산출: 원천 글 모음(JSON).

### ② 발굴 (운영자/구독)
수집 캡션을 **운영자가 직접 읽고** 에이전트/스킬 아이디어 추출. 순수 홍보·동기부여·정치·밈 제외.
(API 스크립트 X)

### ③ 중복성 판정 (내용 기반 · 운영자/구독)
판정 단위는 **이름·키워드가 아니라 기능 내용**(핵심 기능·산출물·입력·도메인·차별점)이다. 각 후보를 당신의
카탈로그(name·tagline·description)에서 **최근접 1~3개**와 직접 대조하고, 핵심 기능/산출물/도메인 항목을
비교해 **내용 겹침률(0~100%)** 을 추정한다.

| 겹침률 | 판정 | 처리 |
|---|---|---|
| **< 50%** (차이 50%+) | **NEW** | 별개 에이전트로 인정 → 제조 큐 진입 |
| **50~79%** | **PARTIAL** | 후보의 차별 기능을 기존에 **흡수(MERGE)**. 차별점이 독립적으로 유용하면 신규 제조도 가능 |
| **≥ 80%** | **DUP** | 신규 제조 안 함. 단 후보에 기존에 없는 좋은 내용이 있으면 기존 에이전트에 **흡수** |

- **흡수(MERGE) 규칙**: DUP/PARTIAL이어도 후보 캡션의 유용한 기능·입력·산출 아이디어는 대상 기존
  에이전트의 `system_prompt`/소개에 반영한다(개선 큐). 중복이라고 버리지 않는다.
- **비제외 원칙**: "리치 도구 필요"·"too-heavy(무거움)"는 **제외 사유가 아니다.** 유용하면 만든다.
  구현 부담은 포털형 인프라 + 전용 실행기(이미지/영상/음성/문서)로 흡수하고, 최종 취사는 ⑥ 80 게이트가
  한다. ③에서 난이도·무게로 컷하지 않는다.

### ④ 인기도 필터 (운영자/구독)
출처 글 인기도(♥) **기본 10 미만 제외.** 수요 신호 바닥권 컷.
→ 산출: 후보 숏리스트(JSON).

### ⑤ 제조 = 에이전트 제조 방법론 (운영자/구독 + 불가피 API)
통과 후보를 에이전트 제조 공정으로 제조:
- **BOM·부품 발굴·검증**: 공개저장소·OSS·모델 허브·사내자산에서 부품 발굴→분해→**안전성·인기도
  (star/download)·유지보수 검증→검증된 부품만 적용.** 저인기·미유지·미검증 부품 배제.
- **정체성 10요소** 명시: 목표·자율루프·도구·메모리·이름·분류·LLM선택·종료·보안·비용한도.
- **구동방식 = 포털형**: 런타임 + DB(system_prompt·도구 ID). 리치 출력 필요 시 전용 실행기 —
  이미지/영상/음성/문서 생성기를 붙인다(불가피 API).
- **중복 금지**(제조 단계 경계 결정), **자기완결**, QC(예: 5기준 평가)는 운영자가 판정.

### ⑥ 평가 → 80 게이트 (운영자/구독)
제조한 에이전트의 라이브 유용성을 **운영자가 직접 1~100 판정**(실물 산출 여부·완성도). **80점 미만 제외.**
통과만 출시. (실제 프로덕션 경로 확인이 꼭 필요할 때만 런타임 API 1~2회.)

### ⑦ 출시 (API: 배포 + 사람 승인 게이트)
통과 에이전트를 플랫폼에 출시:
- DB upsert(status=approved, system_prompt, category/subcategory, **provider는 크레딧 있는 것**)
- 소개 데이터 + 카탈로그 카드·카운트 + 에이전트 페이지
- 배포 (프로덕션 변경 = **사람 승인 게이트**)
- 라이브 검증: 구조(카드·페이지 응답) + **기능(실행 산출, 로그인 세션)** — curl 200 금지(UI 철칙)
- 변경 관리 도구를 쓴다면 등재.

## 기본 파라미터
- months=3 · like_threshold=10 · pass_score=80 · max_manufacture=운영자 지정
- 키워드: 운영자 제공 또는 카탈로그 분류 × 니즈 프레이밍 기본 세트

## 자기검증 회피
80 게이트 판정과 출시 후 검증은 분리. 출시는 사람 승인 필수(되돌리기 어려운 프로덕션 변경 — soft 우선).

## 확장
완전 무인 자동(cron) 원하면, 이 스킬을 핵심으로 **자율 실행 에이전트**를 제조.
(현재는 반자동 — 사람 승인 게이트 유지)
