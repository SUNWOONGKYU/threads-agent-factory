# 방법론 설명서 — Threads Agent Factory

> 실수요 발굴 → 내용 기반 중복 판정 → 에이전트 제조 → 80점 게이트 → 승인 후 출시.
> 이 문서는 7단계 파이프라인과, 그 심장인 **내용 기반 중복 판정 4구간**을 상세히 설명합니다.

---

## 0. 한눈 흐름 (게이트 2곳)

```
① 수집(API) → ② 발굴 → ③ 중복성 판정(NEW/PARTIAL/DUP) → ④ 인기도 필터
   → ⑤ 제조(에이전트 제조 방법론) → ⑥ 80점 게이트 → ⑦ 출시(사람 승인)
```

핵심 철학 세 가지:
1. **수요가 곧 주문서다.** 사람이 아이디어를 쥐어짜지 않는다. 소셜의 실수요가 제조 큐를 채운다.
2. **중복은 버리는 게 아니라 흡수한다.** 겹쳐도 좋은 아이디어는 기존 에이전트를 개선하는 데 쓴다.
3. **추론은 구독으로, API는 불가피한 것만.** 공장 경제성의 토대.

---

## 1. 단계별 상세

### ① 수집 (API)
- 소셜 스크레이퍼(예: Apify의 Threads 검색 액터)로 글을 모은다.
- **키워드 = 사용자 니즈·페인·실활용형.** 예: "업무 자동화", "마케팅 자동화", "AI로 만들었", 도메인+자동화.
- **개발자 도구명은 금지.** 특정 CLI·프로토콜·개발 유행어는 잡음이 많아 제외한다.
- 파라미터: `after`=최신 N개월(기본 3), 인기순 정렬, 한국어 필터(한글 일정 길이 이상), 글 ID 중복 제거.
- 산출: 원천 글 모음(JSON). **이 단계만 API를 쓴다.**

### ② 발굴 (구독 추론 · 직접)
- 수집한 캡션을 **사람이 직접 읽고** 에이전트/스킬 아이디어를 추출한다.
- 순수 홍보·동기부여·정치·밈은 제외. "이런 걸 자동으로 해주면 좋겠다"는 실수요 신호만 남긴다.
- 추론을 API 스크립트로 돌리지 않는다 — 구독 LLM으로 직접 한다.

### ③ 중복성 판정 — 내용 기반 (구독 추론 · 직접) ★ 이 공장의 심장
판정 단위는 **이름·키워드가 아니라 기능 내용**이다. 각 후보를 당신의 카탈로그에서 **최근접 1~3개**와
대조하고, 다음 항목을 비교해 **내용 겹침률(0~100%)** 을 추정한다:

- 핵심 기능 (무엇을 하는가)
- 산출물 (무엇을 내놓는가)
- 입력 (무엇을 받는가)
- 도메인 (어느 분야인가)
- 차별점 (기존과 무엇이 다른가)

→ 4구간 분류는 §2 참조.

### ④ 인기도 필터 (구독 추론 · 직접)
- 원천 글의 반응(♥) **임계값 미만 컷**(기본 ♥10). 수요 신호가 바닥인 후보를 떨군다.
- 산출: 숏리스트(JSON).

### ⑤ 제조 = 에이전트 제조 방법론 (구독 추론 + 불가피 API)
통과 후보를 제조 공정으로 만든다. 핵심 차용:
- **부품 발굴·검증 (BOM)**: 공개 저장소·OSS·모델 허브·사내 자산에서 부품을 발굴 → 분해 →
  **안전성·인기도(star/download)·유지보수 검증 → 검증된 부품만 적용.** 저인기·미유지·미검증 부품 배제.
  (④에서 잡은 인기도 원칙을 부품 수준에서 한 번 더 적용)
- **정체성 10요소** 명시: 목표 · 자율루프 · 도구 · 메모리 · 이름 · 분류 · LLM 선택 · 종료 · 보안 · 비용 한도.
- **구동방식 = 포털형**: 런타임 + DB(system_prompt·도구 ID). 리치 출력이 필요하면 전용 실행기
  (이미지·영상·음성·문서 생성)를 붙인다. 이 부분이 불가피한 API 호출.
- **자기완결 · 자기검증 회피**: 산출물은 별도 검토(예: 5기준 평가)·검증 주체로 출하 검사. dev 추론은 구독으로.
- **중복 금지**: 제조 단계에서도 기존 기능과 겹치면 중단/병합(③과 연계).

### ⑥ 평가 → 80점 게이트 (구독 추론 · 직접)
- 제조한 에이전트의 **라이브 유용성을 1~100으로 직접 판정**(실물 산출 여부·완성도).
- **80점 미만 제외.** 통과만 출시 큐로.
- 실제 프로덕션 경로 확인이 꼭 필요할 때만 런타임 API를 1~2회 쓴다.

### ⑦ 출시 (API · 사람 승인 게이트)
- 통과 에이전트를 플랫폼에 등재: DB upsert(approved) + 소개/카탈로그 카드 + 에이전트 페이지 + 배포.
- **프로덕션 변경 = 사람 승인 게이트.** 되돌리기 어려우므로 자동 출시 금지(soft 우선).
- 라이브 검증: 구조(카드·페이지 응답) + **기능(실행 산출, 로그인 세션)**. `curl 200` 만으로 "동작함" 판정 금지.

## 2. 내용 기반 중복 판정 — 4구간 상세

| 겹침률 | 판정 | 처리 |
|---|---|---|
| **< 50%** (차이 50%+) | **NEW** | 별개 에이전트로 인정 → 제조 큐 진입 |
| **50~79%** | **PARTIAL** | 후보의 차별 기능을 기존에 **흡수(MERGE)**. 차별점이 독립적으로 유용하면 신규 제조도 가능 |
| **≥ 80%** | **DUP** | 신규 제조 안 함. 단 후보에 기존에 없는 좋은 내용이 있으면 기존 에이전트에 **흡수** |

### 흡수(MERGE) 규칙
DUP/PARTIAL이어도 후보 캡션의 유용한 기능·입력·산출 아이디어는 대상 기존 에이전트의
`system_prompt`/소개에 반영한다(개선 큐). **중복이라고 버리지 않는다.**

### 비제외 원칙
"리치 도구 필요"·"too-heavy(무거움)"는 **제외 사유가 아니다.** 유용하면 만든다 — AI가 만드는 데
난이도는 큰 장벽이 아니다. 구현 부담은 포털형 인프라 + 전용 실행기로 흡수하고, 최종 취사는 ⑥ 80점
게이트가 한다. ③에서 난이도·무게로 컷하지 않는다.

## 3. 기본 파라미터

| 파라미터 | 기본값 | 의미 |
|---|---|---|
| `months` | 3 | 수집 기간(최신 N개월) |
| `like_threshold` | 10 | 인기도 컷(♥ 미만 제외) |
| `pass_score` | 80 | 유용성 합격선 |
| `max_manufacture` | 운영자 지정 | 한 회차 제조 상한 |
| `keywords` | 니즈형 세트 | 사용자 제공 또는 도메인 × 니즈 프레이밍 |

---

## 4. 자동화 vs 반자동

- 본 방법론은 **반자동 운영 도구**다. 발굴·판정·평가는 사람이 구독 LLM으로 직접, 출시는 사람 승인.
- 완전 무인(cron)이 필요하면 이 공장을 핵심으로 **자율 실행 에이전트**를 별도 제조한다.
  (단, 프로덕션 출시 자동화는 위험 — 사람 승인 게이트 유지 권장.)
